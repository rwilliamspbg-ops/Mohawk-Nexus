Protocol Roles

The implementation checklist is intentionally narrow. Each protocol has one job, and the demo only becomes more complex if the core flow is already working.

AG-UI Controls the Loop

The agent decides when the interface changes and which interaction state should be shown next.

A2UI Renders the State

The frontend receives structured updates and turns them into a live panel with visible state transitions.

MCP Apps Stay Optional

Only add a small open-ended surface if the core demo is already reliable and visually clear.

Verification Badge

Show an explicit verified or failed state so the judge sees proof, not just text output.

Fallback Static View

Keep a non-agentic version of the same story in case the live protocol layer stalls during the demo.

Judge-Friendly Story

Every visible element should help answer one question: what did the agent do, and why should I trust it?

Implementation Checklist

This is the exact build order for the hackathon. If a step does not move the demo closer to the AG-UI + A2UI story, it can wait.

Phase 1: Story

Update the pitch and hero copy so the demo clearly says AG-UI drives the interaction and A2UI renders the result.

Phase 2: Surface

Replace static summary panels with one live state panel that changes from idle to analyzing to verified.

Phase 3: Runtime

Wire the single interaction loop, keep the payload tiny, and make every change visible on screen.

Phase 4: Proof

Add one explicit proof or verification indicator so the judge sees the evidence trail instead of inferring it.

Demo Flow

The demo should feel like a sequence, not a pile of features: ask for a result, watch the agent respond, render the live UI, then display the verification state.

Step 1: Request

Start with a simple task, such as checking node health or verifying an aggregate.

Step 2: Agent Decision

AG-UI handles the interaction and decides what the next surface state should be.

Step 3: A2UI Render

The frontend receives declarative updates and turns them into a visible live panel.

Step 4: Verification

The proof badge flips to verified, showing the result is backed by more than just a chat response.

Step 5: Optional Flourish

If time remains, show one MCP Apps surface for an editable diagram or secondary visualization.
